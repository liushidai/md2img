## Context

当前系统已经实现单文档 Markdown 编辑器：页面进入后读取 `/api/documents/current`，在前端本地完成 Markdown 预览与 sanitize，并通过手动保存把正文写入 SQLite。这个实现验证了最小编辑闭环，但把“当前文档”作为后端单例写死，且标题完全依赖正文首个一级标题，无法支撑真实的多文档写作后台。

产品文档与长期 DDL 已经更接近多文档模型：`docs/product.md` 以“创建文档 / 编辑文档”为基础流程，`docs/storage-ddl.md` 也将 `title` 视为独立字段。多文档工作台需要把当前原型向长期模型靠拢，但仍保持范围克制，只解决列表、新建、切换、按文档保存这条主链路，不提前引入删除、搜索、登录与图片生成。

## Goals / Non-Goals

**Goals:**
- 将后端从单文档 `current` 接口升级为资源型多文档接口。
- 将前端从单页编辑器升级为带列表与 URL 选中文档的工作台。
- 引入独立 `title` 元数据，并使保存同时覆盖 `title` 与 `markdownContent`。
- 保留现有本地预览与 sanitize 机制，只重构文档资源模型与交互。
- 明确文档切换时的未保存保护，避免多文档场景下误丢草稿。

**Non-Goals:**
- 不实现删除、搜索、标签、文件夹、自动保存、历史版本。
- 不实现登录、多用户隔离或 owner 语义。
- 不实现图片生成、公开 URL、渲染任务与发布配置。
- 不在本次变更中引入复杂的状态缓存或后台草稿同步机制。

## Decisions

### 1. 前端通过 URL 维护当前选中文档
- 决策：当前选中文档由 `/documents/:id` 路由参数决定，前端以 URL 作为唯一真相，后端不再维护 `current` 语义。
- 原因：这样可以让刷新、前进/后退与深链接都自然成立，也避免后端同时维护“资源集合”和“当前单例”两套语义。
- 备选方案：保留 `/api/documents/current` 并在后端保存当前文档。未采用，因为会让多文档模型长期背负兼容包袱。

### 2. 文档模型最小升级为 `id + title + markdownContent + timestamps`
- 决策：本次仅把 `documents` 表升级为多文档所需的最小字段：`id`、`title`、`markdown_content`、`created_at`、`updated_at`。
- 原因：这些字段足以支撑列表、新建、切换与保存；继续引入 `owner_user_id`、`density_preset` 等长期字段会扩大范围。
- 备选方案：直接切换到 `docs/storage-ddl.md` 的全量文档模型。未采用，因为当前尚未进入登录和发布阶段。

### 3. `title` 作为独立元数据存储，不再由正文 H1 唯一推导
- 决策：文档标题独立存储，用于列表展示、顶部栏和文档身份识别；正文中的一级标题仅作为内容结构，不再是唯一标题来源。
- 原因：多文档列表、空白草稿和文档重命名场景都要求稳定的文档名称，不能再依赖正文首个 H1。
- 备选方案：继续从正文推导标题。未采用，因为会让列表管理能力持续被正文结构绑架。

### 4. 资源型接口只提供列表、新建、按 ID 读取与按 ID 更新
- 决策：后端最小接口集为 `GET /api/documents`、`POST /api/documents`、`GET /api/documents/:id`、`PUT /api/documents/:id`。
- 原因：这是多文档闭环所需最小资源面，能覆盖工作台主要动作，又不提前引入删除等复杂交互。
- 备选方案：同步加入删除或搜索接口。未采用，因为会扩散到确认流程、空态处理与权限边界。

### 5. 新建文档立即创建真实记录并跳转
- 决策：点击“新建文档”时，前端直接请求创建文档，拿到真实 `id` 后跳转到新 URL，而不是先在前端创建临时草稿。
- 原因：真实记录能让路由、列表与保存语义保持一致，避免临时 ID、未持久化实体等额外状态。
- 备选方案：前端先生成本地草稿，首次保存时再创建。未采用，因为状态复杂度更高。

### 6. 切换文档时采用“确认放弃修改”而非自动保存
- 决策：当当前文档存在未保存修改时，用户切换到另一篇文档需要显式确认；第一版不做“自动保存后切换”。
- 原因：自动保存会把切换意图与保存意图耦合在一起，并引入保存失败后的恢复问题。
- 备选方案：切换前自动保存。未采用，因为在多文档首版中收益不够高。

## Risks / Trade-offs

- [旧的 `/api/documents/current` 客户端会失效] → 作为显式 breaking change 在 specs 和 proposal 中声明，并在前后端同一变更内一起迁移。
- [引入路由后会增加页面壳层复杂度] → 保持单一工作台页面结构，不同时引入多层导航与路由嵌套。
- [独立 `title` 可能与正文 H1 不一致] → 明确两者语义不同：`title` 属于文档元数据，正文 H1 属于内容结构。
- [列表与编辑态可能出现“未保存标题”和“已保存列表标题”不一致] → 列表默认只反映最近成功保存状态，避免左侧导航过早显示未持久化草稿名称。

## Migration Plan

1. 扩展共享协议，从单文档 `CurrentDocument` 升级到文档列表、详情、创建与更新类型。
2. 升级后端 SQLite `documents` 表结构与查询方式，从固定单例改为资源集合。
3. 重构文档路由为列表、新建、按 ID 读取与按 ID 更新。
4. 在前端引入 `vue-router`，建立工作台壳层与 `/documents/:id` 路由。
5. 将现有编辑器页面拆分为左侧列表 + 右侧编辑器，并接入未保存切换确认。

若需要回滚，可恢复单文档接口与旧页面；数据层面由于 `documents` 表从单例扩展为集合，需要保留已有第一篇文档数据并重新映射为默认文档。

## Open Questions

- 新建文档默认标题是否固定为“未命名文档”，以及是否需要在首版避免同名。
- 当用户直接访问 `/` 时，前端应跳最近更新文档，还是在无文档时自动创建后再跳转，需要在实现时固定一条规则。

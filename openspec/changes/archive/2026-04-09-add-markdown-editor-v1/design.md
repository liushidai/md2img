## Context

当前代码库只实现了前后端 health-check，尚无真实业务模块。项目约束要求前端使用 Vue 3 + Pinia，后端使用 Elysia，前后端共享协议放在 `packages/shared`，并使用 Bun workspace 统一管理依赖。产品文档的长期目标是“私有 Markdown 文档 + 固定公开图片 URL”，但第一版需要先验证单文档编辑、实时预览、手动保存与刷新回显这条最小主链路，不提前引入图片生成、渲染任务和公开访问语义。

第一版还需要处理一个关键安全边界：右侧预览采用前端本地渲染，但 Markdown 允许用户输入原始 HTML，因此预览阶段必须在 HTML 注入页面前完成 sanitize，避免把编辑器预览变成任意脚本执行入口。

## Goals / Non-Goals

**Goals:**
- 提供一个单页单文档编辑器，支持左侧 Markdown 输入、右侧 HTML 安全预览。
- 提供当前文档读取与手动保存接口，并将 Markdown 原文持久化到 SQLite。
- 明确标题、未保存状态、保存失败与加载失败等页面行为规则。
- 让前端、后端与共享层首次形成完整业务闭环，为后续图片生成能力提供稳定起点。

**Non-Goals:**
- 不实现图片生成、公开图片 UUID、渲染队列、过期和删除语义。
- 不实现多文档管理、登录系统、协作编辑、自动保存与历史版本。
- 不保证前端预览与未来服务端图片渲染结果完全一致。

## Decisions

### 1. 以“单文档”替代完整 CRUD
- 决策：第一版只暴露“当前文档”的读取与保存接口，而不是一开始设计多文档列表、新建、切换与删除。
- 原因：当前目标是验证最小业务闭环，而不是搭建完整后台信息架构。单文档可以显著减少路由、状态、表结构与交互复杂度。
- 备选方案：直接实现多文档 CRUD。未采用，因为会提前引入标题字段、列表页、排序与当前文档选择等不必要范围。

### 2. 仅持久化 Markdown 原文，不存 HTML
- 决策：数据库仅保存 Markdown 原文，HTML 预览在前端运行时生成。
- 原因：Markdown 是规范源数据，HTML 是派生结果。只存原文可以避免双写不一致，并保持未来图片渲染链路仍以 Markdown 为输入。
- 备选方案：同时保存 Markdown 与 HTML 缓存。未采用，因为第一版没有性能压力，而且缓存失效策略会增加复杂度。

### 3. 标题从首个非空一级标题推导
- 决策：不在数据库中单独存储 title，页面标题从 Markdown 首个非空一级标题 `# ` 推导；不存在时显示“未命名文档”。
- 原因：当前仅有一个文档，不需要列表展示，也不需要额外标题编辑控件。推导规则更轻，且与 Markdown 写作习惯一致。
- 备选方案：单独存 title 字段。未采用，因为会带来标题与正文一级标题的双重来源问题。

### 4. 预览在前端本地渲染，并在插入页面前执行 sanitize
- 决策：Markdown 在浏览器内解析为 HTML，再执行 sanitize，最终以安全 HTML 渲染到右侧预览区。
- 原因：本地渲染能提供即时反馈，满足编辑器体验；sanitize 能控制允许的 HTML 子集，阻断脚本、事件处理器和危险协议。
- 备选方案 A：把 Markdown 发给后端，由后端生成 HTML 返回。未采用，因为实时交互变重，且当前不需要统一服务端渲染结果。
- 备选方案 B：完全禁用原始 HTML。未采用，因为已明确需要允许原始 HTML 的安全子集进入预览。

### 5. 页面脏状态以“当前内容是否等于最近一次成功保存内容”为准
- 决策：页面维护一份“最近一次成功保存的 Markdown”，当前编辑值与其比较决定是否存在未保存修改。
- 原因：这样可正确支持“改了又改回去”的场景，避免只要输入过就长期显示未保存。
- 备选方案：用布尔标记记录用户是否编辑过。未采用，因为无法表达内容恢复到已保存状态的情况。

### 6. 数据模型先采用最小 documents 表
- 决策：SQLite 第一版只落一个最小 `documents` 表，至少包含 `id`、`markdown_content`、`created_at`、`updated_at`。
- 原因：当前需求只覆盖单文档保存与回显，不需要提前落地 `public_images`、`render_jobs`、`sessions` 等完整产品模型。
- 备选方案：直接套用 `docs/storage-ddl.md` 的全量表结构。未采用，因为会让第一版被未来能力牵制。

## Risks / Trade-offs

- [前端预览与未来图片渲染不一致] → 明确第一版预览只承担编辑辅助语义，后续图片渲染能力再补统一规则。
- [允许原始 HTML 会扩大安全面] → 采用 sanitize 白名单策略，默认移除脚本、事件属性、危险协议与高风险标签。
- [单文档接口后续扩展为多文档时需要调整 API] → 共享协议与服务实现保持 feature/module 边界清晰，后续在保留当前能力的前提下再扩展 CRUD。
- [SQLite 接入会引入新的运行时依赖] → 选择 Bun 兼容、心智负担低的 SQLite 方案，并把数据库访问收敛到 document 模块内部。

## Migration Plan

1. 新增 SQLite 初始化逻辑与最小 documents 表创建流程。
2. 在 `packages/shared` 定义当前文档读取/保存请求与响应类型。
3. 在 `apps/server` 落地 document 模块与 API。
4. 在 `apps/web` 落地 editor feature、页面状态与本地预览。
5. 完成后以单页编辑器替换当前 health-check 展示页面。

本变更为新增能力，不涉及历史数据迁移。若实现中途回滚，可移除新增模块并恢复原有 health 页面；SQLite 文件可删除后重建。

## Open Questions

- 预览白名单中是否允许 `img`、`table`、`style` 等标签或属性，需要在实现前选定具体策略。
- 单文档的固定主键或初始化策略如何约定：首次访问自动创建空文档，还是读取不到时由服务端返回空对象，这需要在实现阶段定一条简单规则。
